## Standard Library and Operating System
Rust defines its standard library as

> The Rust Standard Library is the foundation of portable Rust software, a set of
> minimal and battle-tested shared abstractions for the broader Rust ecosystem. It
> offers core types, like `Vec<T>` and `Option<T>`, library-defined operations on
> language primitives, standard macros, I/O and multithreading, among many other
> things.

This is true for most implementations of other programming languages as well. The
standard library of each languages and their implementation are a collection of code
providing functionalities like
- Implementation of common data structures and algorithms
- Interaction with external system
- Interaction with the operating system

Beside these, standard library of some languages provides extra functionalities for
other common operations like
- Parsing JSON, XML and other common data formats
- Building a web server

The important ones for today are interaction with operating system and interaction with external system (which is generally done through OS as well).

Operating systems have many important responsibilities one of which is providing
client programs with various services that are very difficult or impossible for
client programs to implement themselves. Some of these services include interacting
with hardware, running new programs, communicating with other programs among other things. The os provides these services through system calls. The most common way
OSes implement syscalls is through a software interrupts. The client programs can
invoke that system call using special instruction.

For example, on x86_64 system, `SYSCALL` is used to invoke a syscall.[^1] This instruction has no operands and the required arguments, along with the syscall number (which identifies which syscall to invoke)
is moved to appropriate registers before issuing the `SYSCALL` instruction.

The standard libraries provide nice abstractions over these system calls making it
easier to interface with the OS.

Here are few of the modules Rust's std exposes:
- `std::alloc`: Memory Allocation API
- `std::fs`: Filesystem manipulation operations
- `std::net`: Networking primitives for TCP/UDP communication
- `std::sync`: Useful synchronization primitives

and many, many more.

## What and Why?
So, what exactly is no_std?
It's an attribute that tells rust to not link against the standard library.
![Barry Series: What gif](../images/barry_what.gif)

You might be wondering why would we not want to link against the standard library
and miss out on all these awesome features.

I said before that the standard library provided a nice abstraction on top of the features the operating system provides. Looking at it another way,
the abstractions the standard library provides depend on operating system to function. Not just the abstractions, but collections like `Vec` use
an allocator provided by the operating system.

So, when we write code that can't utilize operating system functionalities because either there is no operating system support or the support from operating
system is not what we want, then we can't link with standard library. This is the case when we are writing our own operating system or hypervisor or when we
are writing code that directly runs on hardware like embedded systems.

Let's do it. Let's write a program that uses the `no_std` attribute and see what we can come up with.

## How?
We are going to see what it takes to write a rust program without using a standard library. We do need to decide what our program will be. We don't want anything fancy
so lets just go for a program that runs forever without doing anything.

First of all, let's create a binary and run it to make sure everything is working.

```bash
cargo new hello_no_std
cd hello_no_std
cargo run
```

This should print "Hello, World".

Now to tell rust to not link against the standard library, add `#![no_std]` at the top of `main.rs`.

If we go ahead and compile it now, we get the following three errors:

```bash
error: cannot find macro `println` in this scope
 --> src/main.rs:4:5
  |
4 |     println!("Hello, world!");
  |     ^^^^^^^

error: `#[panic_handler]` function required, but not found

error: unwinding panics are not supported without std
  |
  = help: using nightly cargo, use -Zbuild-std with panic="abort" to avoid unwinding
  = note: since the core library is usually precompiled with panic="unwind", rebuilding your crate with panic="abort" may not be enough to fix the problem
```

The `println` macro is defined in the standard library so it can't find it now that we are not linking against it.
But that is not a problem for us, we are just making a program that does nothing but loop forever, so we can
just remove that line and replace with an infinite loop: `loop {}`.

That leaves us with two errors. Both are about panics.

So, what exactly is a panic?
Panics are a mechanism in rust to handle unrecoverable errors, such as accessing an array out of bounds.
The default strategy for handling panics is to unwind the stack where Rust walks back through the stack and cleans
up the data. There is another option which immediately aborts the program without any cleaning up.

There is another aspect to panic handling in rust and that is the `panic_handler` attribute.
This attribute is applied to a function to define the behavior of the panics.

The standard library provides panic handler for both strategy: `unwind` and `abort`.[^2]
The default handler (when available) is the `unwind` handler.

So by not linking with standard library, we also need to define our own `panic_handler` function.That is what the second error is talking about.

Let's go ahead and do that. According to the [reference](https://doc.rust-lang.org/reference/panic.html#the-panic_handler-attribute), the `panic_handler` attribute can only be applied to a
function with signature `fn(&PanicInfo) -> !`.

The `PanicInfo` is a structure that provides the function with information about the panic like
the message associated with the panic (the message given to `panic!` macro), the location information where the panic originated.

So, what are we going to do in our panic function and what is that return type?

The `!` is the `never` type which indicates that this `panic` function should never return.
For now we are just going to do nothing and loop forever just like our main function.

```rust
#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
  loop {}
}
```

If we build our program now, we still get the last error about unwinding panics not being supported without std.
Remember that the default strategy for rust is to unwind. The unwind strategy depends on the
functionalities provided by the standard library and the runtime. So we need to tell rust that
we are using the abort strategy.

We can specify it in our `Cargo.toml`.
```toml
[profile.dev]
panic="abort"

[profile.release]
panic="abort"
```

Now, all three errors are accounted for. So we can build and run to see everything wor-

```bash
error: using `fn main` requires the standard library
  |
  = help: use `#![no_main]` to bypass the Rust generated entrypoint and declare a platform specific entrypoint yourself, usually with `#[no_mangle]`
```

![Mr. Bean Confused Face](../images/mrbean_confused_face.jpg)

What do you mean using `fn main` requires standard library?

In a typical rust binary, the `fn main` is the entry point to our program. But it is not the first piece of code that gets run. Far from it. I'm not yet qualified to explain the intricate details of everything that happens before the program actually gets to main. For that you can
watch [this video](https://www.youtube.com/watch?v=q8irLfXwaFM)

The gist of it is that c runtime sets up some stuff like stack initialization and things related to command line arguments then calls rust runtime. The rust runtime in turn sets up some
additional stuff and only then is our main function is called.

Thus, without standard library there are no c and rust runtime to call our main function.

So, what do we do?

The error message is there to help.
It says we should use `#![no_main]` and supply our own entry point, one that doesn't depend on
rust and c runtime.

It also says we should use `#![no_mangle]`. What is that?

The compiler does something called name mangling which is to encode a unique name for symbols
that are used during code generation. It solves the problem of name collision among other things.
Rust enables name mangling for functions by default. Since our entry point should be something
the can be identified by linker, we need to make sure that compiler keeps the name of our function intact.

With that, we have the following code in our `main.rs`.

```rust
#![no_std]
#![no_main]

use core::panic::PanicInfo;

#[unsafe(no_mangle)]
fn main() {
    loop {}
}

#[panic_handler]
fn panic(_info: &PanicInfo) -> ! {
    loop {}
}
```

The actual entry point for our program start at the label `_start`. This `_start` seems to reference the 
`__libc_start_main` at some point. This symbol is probably defined int the c standard library. But since we 
told rust not to link with standard library (which includes c standard library as well), we get this beautiful
linker error.

So, what do we do now?
We could define `__libc_start_main` ourself. The full specification is [here](https://refspecs.linuxbase.org/LSB_3.1.0/LSB-generic/LSB-generic/baselib---libc-start-main-.html)
and here is the function declaration:
```c
int __libc_start_main(
  int *(main) (int, char * *, char * *), 
  int argc, 
  char * * ubp_av, 
  void (*init) (void), 
  void (*fini) (void), 
  void (*rtld_fini) (void), 
  void (* stack_end));
```

I think we should think of a better way. I mean all our problems occurs because `_start` references that symbol. What if we define our own `_start` symbol.
What does `_start` even do?

As it turns out, it does very important stuff like setting up the stack, move `argc` and `argv` to right places, prepare the call to `__libc_start_main` (whatever that is)
among other things. These are all important things but for us they don't really matter now. 

Since our program takes no input, gives no output, performs no calculations, takes no wife, holds no lands, fathers no children, wears no crown, wins no glory, we can
just rename our `main` fn to `_start`.

If we do that, we get another linker error similar to the one above. This time the error is that there are two `_start` symbols one defined in the `Scrt1.o` and 
one defined by use. Since we have no use of `_start` defined by the c runtime library, we need to tell that to the linker. But how?

The [rustc book](https://doc.rust-lang.org/rustc/codegen-options/index.html#link-arg) has us covered.
> On Unix-like targets that use cc as the linker driver, use -Clink-arg=-Wl,$ARG to pass an argument to the actual linker.

Using cargo, we can pass arguments to rustc in `.cargo/config.toml` file.
```toml
[target.'cfg(target_os = "linux")']
rustflags = ["-C", "link-arg=-nostartfiles"]
```

It is finally over. We are done.

Wait a minute! Yes the program _is_ stuck but how do we know it's because of the `loop {}` and not because of some other cause and 
how do we know our `panic_handler` function does get invoked when a panic occurs. And what about -

Okay, okay, I get it. We need some way to confirm that our very empty program does what it's supposed to do.

How do we do that is the question? We got rid of standard library. Remember when I said one of the functionality standard library provides 
is the abstraction over underlying os's syscalls. We may have lost the access to standard library, but we still have syscall. (I am sure there are 
other ways to confirm but I'm going with the easiest one.)

I'm running linux on x86_64 so I can invoke syscall using the `syscall` instruction. You provide the syscall number associated with the syscall 
you wanna call in register `rax` and then `syscall` instruction. The particular syscall may require various arguments as well. You need to load the arguments 
in the appropriate place before the `syscall` instruction. The syscall we are going to use is the `write` syscall. For linux x86_64, it expects following 
arguments in specified registers:
- `rdi`: file descriptor of where we want to write
- `rsi`: pointer to the data you want to write 
- `rdx`: size of data in bytes you want to write 

The syscall number of `write` is 1 so we load that into register `rax`. We want to write to standard output. On unix, when a program is started, the kernel opens three 
files: standard input(0), standard output(1) and standard error(2). Thus, we also load 1 into register `rdi`. Now, we can use a fixed message and pass that to the 
write syscall, but lets not do that. Instead, we will define a function that takes a `&str` and writes that to the stdout using the `write` syscall.

```rust 
fn write(msg: &str) {
    let msg_ptr = msg.as_ptr();
    let len = msg.len();
    unsafe {
        asm!(
            "syscall",
            in("rax") 1,
            in("rdi") 1,
            in("rsi") msg_ptr,
            in("rdx") len,
        )
    }
}
```

I'm not going to explain this code, but it should be pretty self-explanatory. If not, there will be links to inline assembly resources in [References](#references).

With this, we can also check what those methods on `PanicInfo` do.

```rust 
pub extern "C" fn _start() {
    write("_start\n");
    panic!("oh no! something went wrong");
    // loop {}
}

#[panic_handler]
fn handle_panic(info: &PanicInfo) -> ! {
    if let Some(msg) = info.message().as_str() {
        write("panicked with message: ");
        write(msg)
    } else {
        write("panicked");
    }
    if let Some(loc) = info.location() {
        write(" in file: ");
        write(loc.file());
    }
    write("\n");
    loop {
      write("looping in panic");
    }
}
```

I thought we were done with linker error. But it's not standard library or c runtime this time. It's rust core library.

I again quote rust's documentation.

> The Rust Core Library is the dependency-free foundation of The Rust Standard Library. 
> It is the portable glue between the language and its libraries, defining the intrinsic and 
> primitive building blocks of all Rust code. It links to no upstream libraries, no system libraries, and no libc.

If it doesn't link with anything, whats with the linker error. What is `rust_eh_personality`?
The [documentation](https://doc.rust-lang.org/stable/core/index.html) tells us about that too.
It says that it's a `eh_personality` [language item](https://doc.rust-lang.org/unstable-book/language-features/lang-items.html). This language 
item is used by the failure mechanism of the compiler.
This language item is used when unwinding stack during panic.

![Wait a minute!](../images/wait_a_min.gif)

Did we not define our own panic handler and told rust that we want to use `abort` not `unwind` panic handler. So what gives?

We did tell rust to use abort. And it did. But just for our code. And as you can see in error message, the problem originates in the core library.
The issue is that we are using a prebuilt core library, one that is built using `panic=unwind`. This was not a problem before in our program because we did not call 
panic.

So, what now? How do we make the core library use `panic=abort` as well. We can build the core library ourself. Cargo has `build-std` feature that enables it to 
compile the standard library as part of the compilation graph thus using our `panic=abort` defined in `Cargo.toml`. It does require `rust-src` component and 
`nightly` toolchain.

```bash
rustup component add rust-src --toolchain nightly
```

Now we can do 

```bash 
cargo +nightly build -Zbuild-std=core
```
